## 0. 한 줄 요약

> 채널(Slack/Discord/Telegram)에 종속되지 않는 **나만의 self-hosted "AI 워크플로우 UX 실험실"**.
> 에이전트를 직원처럼 데이터로 추가/변경/제거하고, 같은 엔진 위에 **여러 UX 실험을 싸게 갈아끼우며** AI 워크플로우를 학습한다.

## 1. 동기 (왜 만드는가)

검증된 사용자 동기(원본 동기 더미 기준, 중요도 순):

1. **(최상위) 소통 형식 실험.** Slack/Discord/Telegram의 "메시지 스트림" 형식을 뛰어넘는 다양한 UX를 직접 시도. **이게 프로젝트의 존재 이유.**
2. **채널 종속 탈피.** UI/UX를 내가 원하는 대로 바꿀 수 있어야 한다.
3. **학습.** AI-agent / AI-workflow 설계 감각을 잡고, 여러 접근을 직접 시도(예: 커서처럼 코드와 함께 작업). 더불어 **Kotlin 언어 자체**(코루틴·sealed class·`when`·null 안정성)를 익힌다.
4. **에이전트 = 직원.** 봇을 매번 새로 만드는 귀찮음 없이 쉽게 추가/변경/제거.
5. **함께 작업.** 코드도 보고, 이미지도 주고받고, 에이전트끼리 소통.

비-목표(YAGNI):
- 다중 사용자/팀 기능 (혼자 쓴다).
- 상용 수준 인증/권한 (단일 사용자 — 비밀번호+쿠키 또는 Tailscale/Cloudflare Access 뒤).
- 채널(Slack 등) 유지·호환 (의도적으로 버린다).

## 2. 기술 스택 결정 (확정)

| 항목 | 결정 | 근거 |
|---|---|---|
| 언어 / 런타임 | **Kotlin** (전면 재작성), **JDK 25 LTS** | 실무(JVM/Kotlin) 직결 + 이 프로젝트가 코루틴/sealed class 학습에 최적. 기존 Node/TS 자산은 LLM 생성물이라 학습가치 낮아 폐기 부담 없음. JDK는 LTS 고정 |
| 웹 프레임워크 | **Spring Boot + 코루틴** | 사용자가 이미 아는 유일한 프레임워크 → 학습 예산을 "Kotlin 언어 + AI 워크플로우"에 집중. **단, Phase 0은 UI/HTTP 없는 헤드리스 엔진**이라 웹 계층은 스트리밍 뷰(Phase 2)에서 처음 붙는다. SSE는 그때 `suspend` 컨트롤러 + `Flow`로 충분(WebFlux 불요) |
| 하이브리드(2런타임) | **기각** | 두 런타임 운영 복잡도. 딱 하나로 간다 |
| Ktor / Go | **기각** | Ktor=새 언어+새 프레임워크 동시 부담, Go=실무 연결 애매. Kotlin 학습 예산을 분산시키지 않는다 |
| 기존 Node/TS 코드 | **참고용 명세로만 보존** | 포트 인터페이스·오케스트레이션 흐름은 검증된 설계 참고자료 |

기존 자산에서 가져올 검증된 설계(Node/TS 코드 → Kotlin 재구현):
- 포트 인터페이스: `ChatPort` / `LLMPort`(`complete` + `spawnWorkers`) / `MemoryPort`
- claude-cli `stream-json` 이벤트 파싱(`assistant`/`result`/`rate_limit_event`)
- 오케스트레이터의 단발/병렬 판단·분해·머지
- 동시성 제한 워커 풀(`runWithLimit`) → Kotlin 코루틴 + `Semaphore`로 자연스럽게 대체

## 3. 북극성 아키텍처 — 3층 구조 (목적지)

이 절은 **도달하려는 최종 형태**를 그린다. 단, 가운데 층(안정적 이벤트 API)은 **지금 만들지 않는다** — 아래 ⚠️ 참고.

지향하는 키스톤:

> **엔진과 표현(뷰)을 완전히 가르고, 그 사이에 "안정적인 이벤트/명령 API"를 둔다.**
> 새 UX 실험 = 엔진 불변 + **새 뷰 하나 추가**.

> ⚠️ **키스톤은 선반영하지 않고 연기한다.** "안정적 이벤트 API"는 뷰가 **최소 2개**(예: 스트리밍 채팅 + 병렬 보드) 생긴 뒤 그 공통 모양에서 **귀납적으로 추출**한다. 뷰 0~1개 상태에서 지금 설계하면 추측 기반 추상화(YAGNI 위반)다. 그전까지 엔진은 콘솔/테스트가 직접 호출하는 **순수 도메인 API**로 둔다. (근거: §5-1 리스크, 적대적 서브에이전트 검토)

```
┌─ UX 표면 (실험장 · 자유롭게 갈아끼움) ──────────────────────┐
│  ① 채팅 스트림   ② 병렬 진행 보드   ③ 코드 캔버스          │
│  ④ 에이전트 조직도   ⑤ 에이전트끼리 대화 극장   …          │
└───────────────────────────┬────────────────────────────────┘
                            │  안정적인 이벤트/명령 API (SSE/WebSocket)
                            │  "세션·메시지·태스크·에이전트 이벤트가 흐름"
┌─ 엔진 (코어 · 거의 안 바뀜) ────────────────────────────────┐
│  Orchestrator   단발/병렬 판단 · 분해 · 머지                 │
│  Agent Registry 에이전트 = 데이터(이름·역할·허용툴·모델)     │
│  LLM Port       claude-cli ▸ Flow 스트리밍, 폴백 API         │
│  Memory Port    SQLite(세션·메시지·태스크·에이전트·메모리)   │
│  (후순위) Knowledge  카파시 LLM wiki 참조                    │
└─────────────────────────────────────────────────────────────┘
```

이는 기존 헥사고날 설계의 자연스러운 확장이다: 단일 "Slack 어댑터"였던 Chat Port를 **"여러 뷰가 붙는 이벤트 API"**로 키운 것.

### 층별 책임
- **엔진:** 외부 의존성(claude-cli, SQLite)을 어댑터로 격리한 도메인 코어. UX를 전혀 모른다.
- **이벤트/명령 API:** 엔진 상태 변화를 이벤트로 흘리고(아래로는 read 모델), 뷰의 명령을 받는다(위로는 command). **이 계약의 안정성이 "UX 실험을 싸게" 만드는 전제.**
- **UX 표면:** 같은 이벤트 스트림을 서로 다른 방식으로 시각화/조작하는 클라이언트들. 자유롭게 추가·폐기.

## 4. 로드맵 — 건설 순서

각 단계는 다음 단계의 토대다. **1번이 "헤드리스 엔진"인 이유: UI보다 엔진이 먼저다.** UI 없이 콘솔/테스트로 LLM 호출 파이프를 먼저 깔면, 구독 헤드리스라 **비용 0**으로 코어를 다질 수 있다. 그리고 **병렬을 스트리밍보다 앞에 둔다** — 병렬 워커가 이 프로젝트의 첫 차별화(Slack으로 불가능한 것)이자 학습 핵심(코루틴 동시성)이라, 웹/스트리밍 배관보다 먼저 검증한다.

| Phase | 무엇 | 왜 이 순서 | 학습 포인트 |
|---|---|---|---|
| **0. 헤드리스 엔진** | `LlmPort` + claude-cli **스트리밍 어댑터**를 **콘솔/테스트로만** (UI·HTTP 없음). primary claude-cli ▸ 폴백 Anthropic API | 모든 것이 딛고 설 코어. UI 전에 LLM 파이프부터. 구독 헤드리스 → 비용 0 | Kotlin·코루틴·`Flow`·프로세스 spawn·sealed interface + exhaustive `when` |
| **1. 멀티에이전트 병렬** | 오케스트레이터 **병렬 위임**(단발/병렬 판단·분해·머지) + 동시성 제한 워커풀. 여전히 콘솔/테스트 | 이 프로젝트의 첫 차별화이자 코루틴 동시성 학습의 핵심. 웹 배관보다 먼저 | 코루틴 동시성·`Semaphore`·머지 |
| **2. 스트리밍 (첫 뷰)** | claude-cli → `Flow` → **SSE → 브라우저**. 여기서 웹 계층(Spring 코루틴)이 처음 붙는다 | 엔진이 선 뒤 가장 단순한 **첫 뷰**로 "엔진→이벤트→뷰"를 실증 | SSE·`suspend` 컨트롤러·`--include-partial-messages` 토큰 스트리밍 |
| **3. 직원들 = 데이터 + 영속** | **에이전트 = 데이터** 레지스트리(CRUD) + SQLite 영속(세션·메시지·태스크·에이전트) | "쉽게 추가/변경/제거"의 핵심. 역할 주입을 데이터로. 무상태 재구성을 영속으로 뒷받침 | `data class`·`sealed class` 도메인 모델링·영속 |
| **4+. 실험 갤러리** | 병렬 진행 보드·코드 캔버스(커서식)·이미지 주고받기·에이전트끼리 대화 극장 등 | 뷰 2개 이후 키스톤을 추출(§3 ⚠️)하면, 각 실험이 "새 뷰 추가"로 싸진다 | UX 실험 그 자체 |
| **후순위** | **Knowledge Port** (카파시 LLM wiki 참조) | 에이전트가 작업 시 지식 참조 | RAG/지식 주입 |

## 5. 알려진 리스크 / 열린 질문 (검토에서 두들길 지점)

1. **키스톤 과설계 위험:** 채팅과 코드 캔버스는 요구가 극단적으로 다르다. "안정적 이벤트 API" 하나가 정말 둘 다 감당하나? 아니면 YAGNI 위반(미래를 위한 추상화)인가?
2. **학습 목표 vs 배관 작업:** Phase 0의 SSE/프로세스 spawn 배관이 "AI 워크플로우 학습"이라는 본 목적을 늦추진 않나? 더 빨리 "재미있는 실험"에 닿는 경로는 없나?
3. **솔로 토이의 최대 적 = 미완성:** 로드맵이 과하지 않나? Phase 0조차 너무 큰가? 진짜 MVP는 무엇인가?
4. **claude-cli 스트리밍 현실:** `stream-json`의 실제 출력 입자(블록 단위 vs 토큰 델타), `--include-partial-messages` 필요 여부, 병렬 spawn 시 비용/한도. **단정 금지 — 스파이크로 검증할 항목.**
5. **claude-cli 의존 리스크:** CLI 출력 포맷이 바뀌면 파서가 깨진다. 폴백(Anthropic API) 경계는 어디에?
